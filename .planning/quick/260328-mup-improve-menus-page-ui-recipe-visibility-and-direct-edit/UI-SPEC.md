## Interaction Contracts

### MUP-01 — Recipe names visible on list page

**Current:** List cards show "N recipes" count only.
**New:** Cards show a horizontally scrollable pill strip of recipe title_en values directly below the date heading, before the status badge row.

- Data requirement: `menus/page.tsx` must expand the Supabase query from `.select('*, menu_plan_items(id)')` to `.select('*, menu_plan_items(id, recipes(id, title_en, category))')`.
- If `menu_plan_items.length === 0`: render the dashed empty-state text "No recipes yet" in `text-xs` `var(--casa-text-faint)` in place of the pill strip.
- Pill strip is display-only. Tapping a pill does nothing (not a link to recipe detail — too small a target on mobile).
- Recipe count sub-label ("N recipes") is removed; the pill strip replaces it visually. The pill count itself is sufficient context.

### MUP-02 — Inline add/remove on list page (draft menus only)

**Current path:** list → navigate to /menus/[id] → click "Add recipes" → modal → confirm = 5+ interactions.
**New path:** tap "Add recipes" on card → modal opens inline → select → confirm = 3 interactions.

- "Add recipes" Button renders below the pill strip, only when `plan.status === 'draft'`.
- Tapping it opens the existing `RecipePickerModal` in a Dialog, pre-populated with current `selectedRecipeIds` for that plan.
- On modal confirm: call existing `addRecipeToPlan` / `removeRecipeFromPlan` server actions; re-fetch that plan's items; update pill strip in place without full page reload.
- Because `menus/page.tsx` is currently a Server Component, this interaction requires extracting the card interaction into a `'use client'` component (e.g. `MenuCard`). The page itself remains a Server Component that fetches data and passes it as props; `MenuCard` handles local open/close state and mutation.
- Confirmed menus: "Add recipes" button is not rendered. No edit controls appear on confirmed cards. The card is read-only.

### MUP-03 — Recipe removal from list page (draft menus only)

**Current path:** list → navigate to /menus/[id] → find recipe → click X = 4 interactions.
**New path:** "Add recipes" opens picker with current recipes pre-selected → deselect to remove → confirm = 3 interactions.

- Removal is achieved via the same RecipePickerModal: the user deselects a recipe to remove it.
- No separate inline X button is added to list-page pills — pill strip is display-only. This keeps the list card clean on mobile.
- On the detail page (`/menus/[id]`), the existing X button behaviour is preserved for confirmed path users who prefer the detail view.

### MUP-04 — Confirmed menu read-only state on list page

- Confirmed cards: pill strip visible (read-only), no "Add recipes" button, no X buttons on pills.
- Card background: `var(--casa-surface-2)` signals non-editable state.
- An "Edit plan" button label changes to "View plan" for confirmed menus (outline variant, same styling). This button still links to `/menus/[id]` for viewing AI rationale, rules warnings, and the delete action.
- Confirmed cards do not need an unlock/re-edit affordance on the list page — full edits go through the detail page.

### MUP-05 — Loading and optimistic states

- When the inline picker confirm fires: the pill strip shows a skeleton pulse animation (`animate-pulse`, 3 pill-shaped placeholders of `var(--casa-border)` bg) while the mutations run. Duration: until re-fetch completes.
- Success: pill strip updates to new recipe list. No toast needed for add/remove (silent success — the pill strip is the confirmation).
- Error: sonner toast "Couldn't update the menu. Try again." The pill strip reverts to prior state.
- "Adding..." label on "Add recipes" Button while mutation is in-flight (spinner + text; Button is disabled).
