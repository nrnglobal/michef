# Quick Task 260329-018: Improve Menus Page UI per UI-SPEC Summary

**One-liner:** Recipe name pills on list cards with inline add/remove via RecipePickerModal for draft menus, read-only confirmed card styling with softer background and "View plan" copy.

---

## What Was Built

The menus list page (`/menus`) was redesigned from showing a plain recipe count to showing an interactive card per menu plan:

- **Recipe pill strip** (MUP-01): Horizontal scrollable strip of `title_en` pills replaces "N recipes" count. Pills use `var(--casa-surface-2)` background with `var(--casa-border)` border. Empty state shows "No recipes yet" in faint text.
- **Inline add/remove** (MUP-02, MUP-03): Draft cards show "Add recipes" outline button below pill strip. Tapping opens `RecipePickerModal` pre-populated with current recipe IDs. On confirm, mutations run server-side then the card re-fetches and updates in place.
- **Confirmed read-only styling** (MUP-04): Confirmed cards use `var(--casa-surface-2)` background (softer), no "Add recipes" button, and "View plan" button text instead of "Edit plan".
- **Mutation loading state** (MUP-05): While mutations are in-flight, 3 skeleton pill placeholders with `animate-pulse` replace the real pill strip. "Adding..." disabled button with spinner shown on draft cards.
- **Error handling**: On any mutation failure, `toast.error("Couldn't update the menu. Try again.")` fires and local state reverts to pre-mutation items.

## Architecture

```
app/(client)/menus/page.tsx  (Server Component)
  └── MenuCard (client, per plan)
        └── RecipePickerModal (shared @/components/)

app/(client)/menus/[id]/page.tsx (Client Component)
  └── RecipePickerModal (shared @/components/)
```

`menus/page.tsx` remains a Server Component. It now fetches expanded data:
```
menu_plan_items(id, recipe_id, sort_order, recipes(id, title_en, category))
```
and passes each plan as props to `MenuCard`, which owns all client-side interaction.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] TypeScript type mismatch on Supabase re-query result**
- **Found during:** Task 2 build verification
- **Issue:** Supabase `.select()` with nested relation returns `recipes` as an array in inferred types (`{ id, title_en, category }[]`) but our `MenuPlanItem` interface types it as a single object (`{ id, title_en, category } | null`). TypeScript rejected the direct cast.
- **Fix:** Cast through `unknown` first: `data as unknown as MenuPlanItem[]`. The runtime data is actually singular (Supabase returns the first match for a to-one relation), so the cast is safe.
- **Files modified:** `app/(client)/menus/menu-card.tsx` line 70
- **Commit:** `b9f05bd`
