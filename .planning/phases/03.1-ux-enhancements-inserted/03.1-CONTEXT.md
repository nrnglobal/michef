# Phase 3.1: UX Enhancements - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Four areas of improvement to the core client workflow:
1. **`/inventory`** — Full CRUD with category tabs (Fridge/Pantry/Spices), staple star-toggle, staples filter, and "Add to Shopping List" per item
2. **`/shopping-lists`** — Manual add of new items (edit/delete already exists from quick task 260324-ccy)
3. **`/recipes/[id]`** — "Add to Menu" button that opens a draft-menu picker modal
4. **`/menus/[id]`** — Increase recipe limit from 4 → 10, fix dark mode rendering issues, fix date-edit visibility

</domain>

<decisions>
## Implementation Decisions

### Inventory — Category tabs

- **D-01:** Keep all 7 existing DB categories (Produce, Protein, Dairy, Grains, Pantry, Spices, Other) — no migration to categories column needed
- **D-02:** Group categories into 3 tabs in the UI:
  - **Fridge** tab = items with category in (Produce, Protein, Dairy, Grains, Other)
  - **Pantry** tab = items with category = Pantry
  - **Spices** tab = items with category = Spices
- **D-03:** "Show All" option appears alongside the 3 tabs to display every item regardless of category
- **D-04:** When adding/editing an item, the category dropdown still exposes all 7 values (no change to input options)

### Inventory — Staples

- **D-05:** Add `is_staple BOOLEAN DEFAULT false` column to `fridge_staples` table via migration (`006_add_is_staple.sql`)
- **D-06:** Each inventory item row shows a star icon (outline = not staple, filled = staple); clicking it toggles `is_staple` via a Supabase UPDATE — no confirmation needed
- **D-07:** A "Staples" filter button appears in the header (alongside or near the tab bar); when active it narrows the visible items to those where `is_staple = true` across all tabs
- **D-08:** Staple items show a filled star icon as a visual badge regardless of filter state

### Inventory — "Add to Shopping List"

- **D-09:** Adds the item to the **upcoming confirmed menu plan's shopping list** (same query as `/shopping-lists/page.tsx` uses — find next confirmed plan by visit_date ≥ today, ascending)
- **D-10:** If no confirmed shopping list exists, show an inline error message: "No upcoming shopping list — confirm a menu first"
- **D-11:** The item is inserted into `shopping_list_items` with: `ingredient_name_en = item_name_en`, `ingredient_name_es = item_name_es`, `quantity = null`, `unit = null`, `category = staple`, `is_always_stock = is_staple` (carry the staple flag through)
- **D-12:** After adding, show a brief success toast/inline confirmation ("Added to shopping list")

### Shopping list — manual add

- **D-13:** Shopping list already supports edit/delete from quick task 260324-ccy; this phase adds the **add-new-item** capability
- **D-14:** Add form fields: `ingredient_name_en` (required), `quantity` (optional numeric), `unit` (optional text), `category` (optional dropdown using the existing category list)
- **D-15:** Spanish name defaults to English value if left blank (matches bilingual pattern across the app)

### Recipe detail — "Add to Menu"

- **D-16:** Add an "Add to Menu" button to `app/(client)/recipes/[id]/page.tsx` (or its `RecipeDetailActions` client component); this is a client-side interaction so it lives in `RecipeDetailActions`
- **D-17:** Clicking opens a modal listing upcoming **draft** menu plans ordered by visit_date ascending; each row shows the date and current recipe count
- **D-18:** Selecting a menu adds the recipe via `addRecipeToMenuPlan()` (new server action in `lib/menu-actions.ts`) which inserts a `menu_plan_items` row; if the recipe is already on that plan, show a message "Already on this menu"
- **D-19:** If no draft menus exist, the modal shows "No draft menus — create one first" with a link to `/menus`
- **D-20:** After successful add, show brief confirmation ("Added to [date] menu") and close the modal

### Menu editor — recipe limit

- **D-21:** Increase max recipes from 4 → 10 in `RecipePickerModal` and in the `confirmMenuPlan` validation in `lib/menu-actions.ts`
- **D-22:** Minimum stays at 2 recipes before confirming
- **D-23:** The "Add Recipe" button should remain clickable until 10 items are reached (currently it visually disables at 4)

### Menu editor — date editing

- **D-24:** The date edit UI already exists (`isEditingDate` state, `dateInput`, `updateMenuPlanDate` action) but is apparently invisible in current state — investigate why and ensure the date field is always visible with clear affordance to tap/click it
- **D-25:** Date field should show the formatted visit date with a pencil/edit icon; clicking anywhere on the date row enters edit mode

### Menu editor — dark mode

- **D-26:** Audit the menu detail page (`app/(client)/menus/[id]/page.tsx`) for any hardcoded hex colours or Tailwind colour classes (e.g., `text-black`, `bg-white`, `text-gray-*`) and replace with `var(--casa-*)` CSS custom properties following the established pattern
- **D-27:** Focus on the main heading (confirmed as black-on-dark) and any other elements reported as invisible

### Claude's Discretion

- Loading/spinner states during "Add to Shopping List" and "Add to Menu" actions
- Exact modal styling for the "Add to Menu" draft-menu picker (follow existing modal patterns)
- Category grouping logic for the Fridge tab (treat anything not Pantry/Spices as Fridge)
- Toast/confirmation implementation (inline message is fine, no library needed)

</decisions>

## Existing Code Insights

### Reusable Assets
- `inventory-client.tsx` — existing inline edit/delete/toggle pattern; extend for staple star-toggle and "Add to Shopping List"
- `shopping-list-client.tsx` — existing edit/delete pattern; add the add-form from the same component

- `lib/menu-actions.ts` — contains `confirmMenuPlan`, `deleteMenuPlan`, `updateMenuPlanDate`; add `addRecipeToMenuPlan` here
- `toTitleCase`, `formatDateEs` in `lib/utils.ts` — use for display strings

### Established Patterns
- Server component fetches → passes data to client component → client manages local state → `router.refresh()` on mutations
- Bilingual: `_en` required, `_es` defaults to `_en` if empty on save
- CSS custom properties: all colours via `var(--casa-*)`, never hardcoded hex

### Integration Points
- "Add to Shopping List": needs the same `shopping_list_id` lookup used in `shopping-list-client.tsx` — find next confirmed menu plan's list
- "Add to Menu": new server action in `lib/menu-actions.ts`; needs to query draft `menu_plans` then insert into `menu_plan_items`
- Migration 006: `ALTER TABLE fridge_staples ADD COLUMN IF NOT EXISTS is_staple BOOLEAN DEFAULT false`
