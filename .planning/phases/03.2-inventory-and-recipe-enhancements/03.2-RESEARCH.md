## Summary

Phase 03.2 is a set of six targeted enhancements that build directly on existing patterns in the codebase rather than introducing new infrastructure. The largest work items are the Google Sheets import feature and the autocomplete database JSON. All other items are surgical changes to existing components.

The most important pre-condition to understand: the one-letter-at-a-time input bug (UX-10, D-08) is definitively caused by `ItemForm` being defined as a function component *inside* the `InventoryClient` function body. React creates a new component type on every render, which forces unmount+remount of the form on every keystroke. The fix is to hoist `ItemForm` to module scope (outside `InventoryClient`) and pass all required data as props.

All image upload work for UX-07 can copy the `resizeImageToLimit` function verbatim from `components/receipt-upload.tsx` — that function already exists and works. The Supabase Storage bucket name is `receipts` (confirmed from the receipt-upload source). The decision (D-04) is to reuse that bucket; recipe images should be stored under a `recipe-images/{recipe_id}/` path prefix to separate them from receipt uploads.

The translate endpoint `/api/ai/translate-recipe` currently expects `title_en`, `description_en`, `ingredients`, `instructions_en`. For inventory single-item translation, the planner should either: (a) call the existing endpoint with only `title_en` set (all other fields empty/absent) and read `title_es` from the response — this works because the endpoint uses `?? ''` for missing fields; or (b) create a minimal `POST /api/ai/translate-item` route. Option (a) is simpler and reuses the working endpoint.

**Primary recommendation:** Fix the input bug first (UX-10) as it affects every other inventory form change. Sequence remaining work: migration → type update → ItemForm extension (source/brand/translate/autocomplete) → Google Sheets import → recipe image upload → add-to-menu filter change.

### Pattern 1: Fixing the ItemForm re-mount bug (UX-10)

**What:** `ItemForm` is currently a function component defined *inside* `InventoryClient`'s function body. React treats it as a new component type on every render, unmounting and remounting the DOM subtly, which resets controlled inputs.

**Root cause (confirmed by reading the code):** Lines 222-320 of `inventory-client.tsx` show `function ItemForm(...)` defined inside `export function InventoryClient(...)`. This is the textbook React pitfall for this exact symptom.

**Fix:** Move `ItemForm` to module scope — above the `InventoryClient` declaration — and pass all needed values (`saving`, `inputStyle`) as explicit props.

**When to use:** Any time a child component definition lives inside a parent function body.

```typescript
// BEFORE (line 222 in inventory-client.tsx) — inside InventoryClient body:
function ItemForm({ form, onChange, onSubmit, onCancel, submitLabel }) { ... }

// AFTER — at module scope, before InventoryClient:
interface ItemFormProps {
  form: FormState
  onChange: (f: FormState) => void
  onSubmit: (e: React.FormEvent) => void
  onCancel: () => void
  submitLabel: string
  saving: boolean
  suggestions?: GroceryItem[]
  onSuggestionSelect?: (item: GroceryItem) => void
  onTranslate?: () => void
  translating?: boolean
}

function ItemForm({ form, onChange, onSubmit, onCancel, submitLabel, saving, ... }: ItemFormProps) {
  // same JSX as before
}
```

async function handleImageUpload(file: File) {

  const supabase = createClient()
  const path = `recipe-images/${recipeId ?? 'new'}/${Date.now()}.jpg`

    .from('receipts')

### Pattern 4: Google Sheets CSV import (UX-06)

**What:** Extract spreadsheet ID from the shared URL, fetch CSV via the public export endpoint, parse CSV rows, match against existing items by `item_name_en`.

```typescript
// URL patterns to handle:
// https://docs.google.com/spreadsheets/d/{ID}/edit#gid=0
// https://docs.google.com/spreadsheets/d/{ID}/edit?usp=sharing
// https://docs.google.com/spreadsheets/d/{ID}/pub?...

function extractSheetId(url: string): string | null {
  const match = url.match(/\/spreadsheets\/d\/([a-zA-Z0-9-_]+)/)
  return match ? match[1] : null
}

function buildCsvExportUrl(sheetId: string): string {
  return `https://docs.google.com/spreadsheets/d/${sheetId}/export?format=csv`
}

// Fetch must go through a Next.js API route — browser cannot fetch
// docs.google.com directly due to CORS
```

**CRITICAL:** The CSV fetch **cannot be done client-side** due to CORS. The Google Sheets export URL responds with `Access-Control-Allow-Origin: *` only for some sheets. In practice, browser fetches to `docs.google.com` fail with CORS errors. The fetch must be proxied through a Next.js API route (e.g., `POST /api/inventory/import-sheets`).

The API route receives the sheet URL, extracts the ID, fetches the CSV server-side (no CORS restrictions), and returns the parsed rows as JSON. The client component then handles the duplicate-resolution UI.

**CSV parsing (manual — no library):**
```typescript
function parseCsv(text: string): Record<string, string>[] {
  const lines = text.trim().split('\n')
  const headers = lines[0].split(',').map(h => h.trim().replace(/^"|"$/g, ''))
  return lines.slice(1).map(line => {
    const values = line.split(',').map(v => v.trim().replace(/^"|"$/g, ''))
    return Object.fromEntries(headers.map((h, i) => [h, values[i] ?? '']))
  })
}
```

Expected columns (from template D-20): `item_name_en`, `item_name_es`, `quantity`, `category`, `source`, `brand`, `is_staple`.

### Anti-Patterns to Avoid

- **Defining components inside render functions:** The root cause of UX-10. Never define a named function component (`function Foo() {}`) inside another component's body. Always define at module scope.
- **Fetching external URLs client-side that need CORS:** Google Sheets export URLs require a server-side proxy — do not `fetch` them directly from the browser.
- **Uploading before recipe ID exists:** For new recipes, generate a temp client UUID for the storage path, then save the recipe with the resulting `image_url`. Don't upload to a path without an identifier.
- **Blocking the save button during image upload:** Let the user fill the form while the image uploads in the background; only block on the final save if the upload is still in progress.
- **Hard-coding the Supabase bucket name:** Use a constant. The bucket is `receipts` (confirmed from `receipt-upload.tsx` line 74).

## Common Pitfalls

### Pitfall 1: ItemForm recreated on every render (the bug itself)

**What goes wrong:** Inputs lose focus after every keystroke; typing requires clicking back into the field after each character.

**Why it happens:** `ItemForm` is defined inside `InventoryClient`'s function body. React sees a new function reference on each render and interprets it as a different component, triggering unmount+remount of the entire subtree.

**How to avoid:** Define `ItemForm` at module scope (before `InventoryClient`). Pass `saving` and `inputStyle` as props.

**Warning signs:** Any input inside an inline-defined child component that loses focus on change.

### Pitfall 2: CORS on Google Sheets fetch

**What goes wrong:** `fetch('https://docs.google.com/spreadsheets/d/{ID}/export?format=csv')` fails in the browser with a CORS error even if the sheet is publicly shared.

**Why it happens:** Google Sheets does not set permissive CORS headers on all export requests when called from arbitrary browser origins.

**How to avoid:** Always proxy through a Next.js API route (`/api/inventory/import-sheets`). The route handler runs on the server where CORS doesn't apply.

**Warning signs:** Network tab shows CORS preflight failure to `docs.google.com`.

### Pitfall 3: Image upload timing on new recipe create

**What goes wrong:** Recipe is saved without `image_url` because the image upload hasn't completed, or the recipe save fails because there's no `id` yet to use as the storage path.

**Why it happens:** The recipe `id` is generated by Supabase (UUID) at insert time; it's not known before the insert.

**How to avoid:** Generate a `crypto.randomUUID()` client-side as a temp path key before any upload. Use it as the path prefix: `recipe-images/${tempKey}/${Date.now()}.jpg`. The resulting public URL is stored in `image_url` regardless of whether the recipe ID matches the path.

**Warning signs:** Recipe is saved with `image_url: ''` even though the user selected a file.

### Pitfall 4: Autocomplete dropdown positioned incorrectly on mobile

**What goes wrong:** The suggestions dropdown renders underneath the keyboard or is clipped by the parent container's `overflow: hidden`.

**How to avoid:** Position the dropdown with `position: absolute` and ensure the parent `ItemForm` container does not have `overflow: hidden`. Set a reasonable `max-height` with `overflow-y: auto` on the suggestions list.

### Pitfall 5: Google Sheets CSV columns misaligned if template is modified

**What goes wrong:** User modifies column order in the template; import silently assigns wrong values.

**Why it happens:** If parsing is position-based (split by index), column order changes break it.

**How to avoid:** Always parse by header name, not by column index. The CSV parse function should build a `Record<header, value>` map per row.

### Pitfall 6: translate-recipe endpoint called for bulk import with many blank items

**What goes wrong:** Importing a 50-row sheet with 30 blank Spanish names fires 30 serial translation calls; UI hangs for 30+ seconds.

**Why it happens:** Each translate call is awaited in sequence inside a loop.

**How to avoid:** Batch all blank Spanish names into a single Claude call. The existing endpoint can translate an array of names if the prompt is adjusted. Alternatively, send all items needing translation in one call by constructing a prompt that returns a JSON array.

### resizeImageToLimit (copy verbatim from receipt-upload.tsx)

```typescript
// Source: components/receipt-upload.tsx lines 21-41
async function resizeImageToLimit(file: File, maxBytes = 5 * 1024 * 1024): Promise<Blob> {
  return new Promise((resolve) => {
    const img = new Image()
    const url = URL.createObjectURL(file)
    img.onload = () => {
      let { width, height } = img
      const scale = Math.sqrt(maxBytes / file.size)
      if (scale < 1) {
        width = Math.floor(width * scale)
        height = Math.floor(height * scale)
      }
      const canvas = document.createElement('canvas')
      canvas.width = width
      canvas.height = height
      canvas.getContext('2d')!.drawImage(img, 0, 0, width, height)
      URL.revokeObjectURL(url)
      canvas.toBlob((blob) => resolve(blob!), 'image/jpeg', 0.85)
    }
    img.src = url
  })
}
```

## Open Questions

1. **New recipe image upload — ID timing**
   - What we know: Supabase generates the recipe UUID at INSERT time; we don't have it client-side before save
   - What's unclear: Whether to use a client-generated temp UUID or defer the upload until after save
   - Recommendation: Generate `crypto.randomUUID()` client-side before form mount; use it as the storage path prefix. The URL is stored in `image_url` at save time. This avoids a two-step save flow.

2. **Google Sheets bulk translate for blank Spanish names**
   - What we know: The existing translate endpoint handles one item's `title_en` per call; calling it 30 times in a loop for a large import would be slow
   - What's unclear: Whether the import use case needs a batched translate endpoint or whether serial calls are acceptable for typical import sizes (< 20 blank items)
   - Recommendation: For the first implementation, batch all blank Spanish names into a single Claude call using a simple comma-separated list prompt. This avoids both the serial-call problem and the need for a new endpoint.

3. **Supabase Storage bucket — receipts bucket for recipe images**
   - What we know: D-04 says to reuse the existing `receipts` bucket; `receipt-upload.tsx` confirms the bucket name is `receipts`
   - What's unclear: Whether RLS on the `receipts` bucket permits authenticated writes without restriction (it's used by the cook for receipts, and now the client for recipe images)
   - Recommendation: The `receipts` bucket is already used by authenticated users; the existing RLS policy should permit the client to upload recipe images. Verify this during Wave 0 of implementation.
